## 2. Git分支策略

### 2.1 分支模型 (Git Flow)

采用Git Flow分支模型，结合团队开发特点进行优化。

```
master (main)
  ↑
  │     hotfix/*
  │     ↑     ↓
  │  ←───────┘
  │
develop
  ↑
  │     feature/*
  │     ↑     ↓
  │  ←───────┘
  │
  │     release/*
  │     ↑     ↓
  │  ←───────┘
```

### 2.2 分支说明

#### 主分支 (Protected)

**master/main分支**
- 生产环境代码
- 只允许通过PR合并
- 每个提交都应该是可部署的
- 需要CI/CD通过和代码审查
- 版本标签在此分支创建

**develop分支**
- 开发主分支，集成所有功能
- 下一个发布版本的基础
- 需要CI/CD通过
- 每日自动构建和部署到开发环境

#### 功能分支

**feature/*分支**
- 新功能开发分支
- 从develop分支创建
- 命名规范: `feature/功能名称` 或 `feature/JIRA-123-功能名称`
- 完成后合并回develop分支
- 示例:
  - `feature/user-authentication`
  - `feature/AI-123-calendar-view`
  - `feature/wechat-login-integration`

**hotfix/*分支**
- 生产环境紧急修复
- 从master分支创建
- 命名规范: `hotfix/修复描述` 或 `hotfix/JIRA-456-修复描述`
- 需要同时合并到master和develop
- 示例:
  - `hotfix/fix-login-error`
  - `hotfix/AI-456-memory-leak`

**release/*分支**
- 发布准备分支
- 从develop分支创建
- 命名规范: `release/版本号`
- 用于测试和修复bug
- 完成后合并到master和develop
- 示例:
  - `release/v1.0.0`
  - `release/v1.1.0`

#### 个人分支

**personal/*分支**
- 个人实验或学习用途
- 不会合并到主分支
- 命名规范: `personal/姓名/功能`
- 示例:
  - `personal/zhangsan/try-new-feature`
  - `personal/lisi/experiment-ui`

### 2.3 分支命名规范

#### 功能分支命名
```
feature/{JIRA-ID}-{brief-description}
feature/{brief-description}

示例:
feature/AI-123-wechat-login
feature/calendar-view-enhancement
```

#### 修复分支命名
```
hotfix/{JIRA-ID}-{brief-description}
hotfix/{brief-description}

示例:
hotfix/AI-456-fix-memory-leak
hotfix/update-dependency-version
```

#### 发布分支命名
```
release/{version}

示例:
release/v1.0.0
release/v1.1.0-beta.1
```

## 3. 提交规范 (Conventional Commits)

### 3.1 提交格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 3.2 提交类型 (Type)

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat: add wechat login |
| fix | 修复bug | fix: resolve memory leak |
| docs | 文档更新 | docs: update API documentation |
| style | 代码格式 | style: format code with prettier |
| refactor | 代码重构 | refactor: simplify authentication logic |
| test | 测试相关 | test: add unit tests for login |
| chore | 构建/工具 | chore: update dependencies |
| perf | 性能优化 | perf: optimize database queries |
| ci | CI/CD相关 | ci: update GitHub Actions |
| build | 构建相关 | build: update webpack config |
| revert | 回滚代码 | revert: revert previous commit |

### 3.3 提交示例

#### 简单提交
```bash
git commit -m "feat: add calendar view component"
git commit -m "fix: resolve event creation validation error"
git commit -m "docs: update API documentation for auth endpoints"
```

#### 详细提交
```bash

- Add wechat login SDK integration
- Implement QR code generation and validation
- Add JWT token management
- Include error handling and user feedback

Closes AI-123"
```

#### 多行提交

## 5. 版本管理

### 5.1 版本号规范 (Semantic Versioning)

版本格式: `主版本号.次版本号.修订号` (MAJOR.MINOR.PATCH)

- **MAJOR**: 不兼容的API变更
- **MINOR**: 向下兼容的功能新增
- **PATCH**: 向下兼容的问题修正

### 5.2 版本发布流程

1. **创建发布分支**
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b release/v1.0.0
   ```

2. **更新版本号**
   ```bash
   npm version patch  # 1.0.0 -> 1.0.1
   npm version minor  # 1.0.0 -> 1.1.0
   npm version major  # 1.0.0 -> 2.0.0
   ```

3. **测试和修复**
   - 在release分支进行测试
   - 修复发现的bug
   - 更新CHANGELOG.md

4. **合并到主分支**
   ```bash
   git checkout master
   git merge --no-ff release/v1.0.0
   git tag v1.0.0
   git push origin master --tags
   ```

5. **合并回develop分支**
   ```bash
   git checkout develop
   git merge --no-ff release/v1.0.0
   git push origin develop
   ```

## 7. 开发工作流

### 7.1 日常开发流程

1. **同步主分支**
   ```bash
   git checkout develop
   git pull origin develop
   ```

2. **创建功能分支**
   ```bash
   git checkout -b feature/AI-123-new-feature
   ```

3. **开发和提交**
   ```bash
   # 开发代码
   git add .
   git commit -m "feat: add new feature"
   ```

4. **推送和创建PR**
   ```bash
   git push origin feature/AI-123-new-feature
   # 在GitHub上创建PR
   ```

5. **代码审查和合并**
   - 等待代码审查
   - 根据反馈修改
   - 审查通过后合并

### 7.2 发布流程

1. **创建发布分支**
   ```bash
   git checkout develop
   git pull origin develop
   git checkout -b release/v1.0.0
   ```

2. **版本更新和测试**
   ```bash
   npm version minor
   npm run test
   npm run build
   ```

3. **修复和准备**
   - 修复测试发现的问题
   - 更新文档和变更日志
   - 准备发布说明

4. **合并和标记**
   ```bash
   git checkout master
   git merge --no-ff release/v1.0.0
   git tag v1.0.0
   git push origin master --tags
   ```

5. **部署和监控**
   - 部署到生产环境
   - 监控应用状态
   - 收集用户反馈
